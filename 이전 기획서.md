# 테니스 클럽 시스템 서버 이전 기획서

## 1. 목적 및 배경
- 로컬 환경(localStorage + 상태 파일)에서는 복수 인원, 복수 기기의 동시 사용이 어렵고 데이터 유실 위험이 있습니다.  
- **Ubuntu 24.04 + MariaDB 11.8 + Node.js** 조합을 도입하면
  * MariaDB가 트랜잭션/Revision 기반으로 모든 경기/점수 기록을 영속화하므로 서버 재기동 이후에도 데이터가 유지됩니다.
  * API 서버(`server.js`)를 통해 모든 클라이언트가 **동일한 상태**를 fetch/save해서 충돌 가능성을 줄입니다.
  * Docker를 도입하면 운영 중 버전 차이 없이 이미지화된 스택을 배포·스케일링할 수 있습니다.

## 2. 서버 사양/환경
- 운영체제: Ubuntu 24.04 (LTS) — 안정적인 패키지와 장기 지원.  
- 필수 설치
  * Node.js ≥18 + npm: 애플리케이션 실행 및 dependency 관리용.
  * MariaDB 11.8: 기존 `state.json` 구조를 JSON 컬럼으로 저장하며 인덱싱/백업 가능.
  * Docker/Docker Compose: 향후 컨테이너 배포, 버전 잠금, 마이크로서비스 확장 준비.
- 디렉토리 예시
  * `/opt/tennis-club`: 코드/보조 스크립트
  * `/opt/tennis-club/logs`: `pm2`/systemd 로그 + 백업 스크립트 결과물
  * `/opt/tennis-club/data`: `mysqldump`, `state.json` 백업
  * `/etc/systemd/system/tennis-club.service`: 서비스 등록 시 단일 진입점

## 4. 배포/이전 절차
1. **서버 초기화**
   - `apt update && apt install -y curl git`로 패키지 기준을 최신화하고 필수 툴 설치.
   - **nvm**으로 Node.js 20 설치: 시스템 기본 Node보다 버전 간섭이 적고, 추후 업그레이드/롤백이 쉬움.
   - `npm install -g pm2`는 서비스 재시작/로그 관리를 자동화하고, `pm2` 제공하는 모니터링/프로세스 복구 기능 확보.
2. **소스 코드 배포**
   - `/opt/tennis-club`에 `git clone` 또는 압축 해제. 디렉토리 퍼미션은 `ubuntu` 사용자 소유로 설정하여 security 유지.
   - `config.js`에서 `TENNIS_API_BASE`는 빈 문자열로 두어 **동일 서버/도메인에서 API 호출**하게 구성.
   - `npm install`로 의존성 설치. 설치 실패 시 MariaDB 드라이버(mariadb) 설치가 누락됐는지 확인.
3. **MariaDB 설정**
   - MariaDB 11.8은 `JSON` 타입/`JSON_EXTRACT` 지원이 있어 `tennis_state.data`를 그대로 저장할 수 있음.
   - 다음 SQL로 DB/사용자 생성: (보안상 `tennis` 사용자만 이용, root는 내부에서만 사용)
     ```sql
     CREATE DATABASE IF NOT EXISTS tennis_club CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
     CREATE USER IF NOT EXISTS 'tennis'@'%' IDENTIFIED BY '<strong_password>';
     GRANT ALL PRIVILEGES ON tennis_club.* TO 'tennis'@'%';
     FLUSH PRIVILEGES;
     ```
   - `/etc/mysql/mariadb.conf.d/50-server.cnf`에서 `bind-address=0.0.0.0`, `max_connections=250`, `innodb_buffer_pool_size=1G` 등을 조정하여 외부 접속, 동시성 확보.
4. **기존 localStorage 데이터 이관**
   - 브라우저 로컬에서 **“로컬 데이터 서버 업로드”** 버튼을 눌러 localStorage → MariaDB. 이유: 자동으로 revision을 포함한 JSON이 서버로 올라가 초기 상태를 동일하게 유지.
   - 백업이 있다면 `curl -X POST http://localhost:3000/api/state`로 `data/state.json`을 서버로 업로드.
5. **서비스 실행**
   - `pm2 start server.js --name tennis-club --watch`로 로그/재시작 자동화. `pm2 save` → `pm2 startup`으로 재부팅 후 자동 복구.
   - alternative: `systemd` 유닛(아래) 사용. 이유: `systemd`는 종속성(MariaDB) 이후 기동, 서비스 재시작 정책, 로그 관리를 제공.
     ```
     [Unit]
     Description=Tennis Club API
     After=network.target mariadb.service

     [Service]
     WorkingDirectory=/opt/tennis-club
     ExecStart=/usr/bin/env node server.js
     Restart=on-failure
     Environment=PORT=3000 DB_HOST=127.0.0.1 DB_NAME=tennis_club DB_USER=tennis DB_PASSWORD=<pwd>

     [Install]
     WantedBy=multi-user.target
     ```
6. **Docker(선택적 컨테이너화)**
   - 이유: 개발/테스트와 운영 환경을 일관되게 유지하려면 앱 + MariaDB를 컨테이너로 묶는 것이 좋음.
   - `Dockerfile`은 Node.js 20 기반 + 소스 복사/`npm install` 포함, `docker-compose.yml`로 서비스 + DB 연동.
   - MariaDB 데이터는 `volumes`로 `/var/lib/mysql`에 유지하고, 환경변수로 자격증명을 주입하여 secrets를 코드에서 빼냄.

-## 5. 설정 파일 및 세부 설정 이유
- Node.js 앱, DB 연결, 백업, Docker 설정 등에서 왜 각 파일이 필요한지 함축 설명.
1. **`server.js`**
   - MariaDB 풀, `/api/state` GET/POST, revision 체크 기능 내장.  
   - 이유: 클라이언트가 모든 오퍼레이션을 HTTP 통신으로 처리하고, Revision을 통해 충돌 감지/해결하므로 중앙 로직이 여기 한 곳에 있어야 유지보수가 수월합니다.
2. **`config.js`**
   - `window.TENNIS_API_BASE`: API가 동일한 호스트/도메인에서 제공될 때 빈 문자열(또는 절대 URL).  
   - 이유: 다중 환경(개발/운영) 구분 없이 같은 스크립트에서 서버 API를 참조하기 위한 구조.
3. **`.env` (선택 사항)**
   - `PORT=3000`, `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_PORT`.  
   - 이유: 배포 시 환경별/운영자별 비밀번호와 포트를 코드에 직접 쓰지 않기 위함.
4. **`docker-compose.yml`**
   - app + MariaDB 서비스 정의, 볼륨, 시크릿 주입.  
   - 이유: Docker Compose를 쓰면 개발자 머신에서도 같은 네트워크/환경으로 테스트한 뒤 그대로 운영할 수 있습니다.
5. **`systemd` 단위 파일**
   - `After=mariadb.service` 설정으로 DB 준비 후 기동, `Restart=on-failure`로 재시작 보장.  
   - 이유: 서버가 재부팅되어도 서비스가 자동으로 올라오고, PM2 없이도 서비스를 관리할 수 있는 표준 방법입니다.
- `server.js`: MariaDB 풀, `/api/state` GET/POST, revision 체크 포인트.
- `config.js`: `window.TENNIS_API_BASE` 정의, 서버 주소 또는 빈 값 유지.
- `.env` (if used): `PORT=3000`, `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_PORT`.
- `docker-compose.yml` 예시:
  ```yaml
  version: '3.8'
  services:
    app:
      build: .
      ports:
        - '3000:3000'
      environment:
        - PORT=3000
        - DB_HOST=db
        - DB_NAME=tennis_club
        - DB_USER=tennis
        - DB_PASSWORD=<pwd>
      depends_on:
        - db
    db:
      image: mariadb:11.8
      restart: unless-stopped
      environment:
        - MYSQL_ROOT_PASSWORD=740923aa
        - MYSQL_DATABASE=tennis_club
        - MYSQL_USER=tennis
        - MYSQL_PASSWORD=<pwd>
      volumes:
        - tennis_mysql:/var/lib/mysql
  volumes:
    tennis_mysql:
  ```

-## 6. 모니터링·백업
- MariaDB 백업: `mysqldump tennis_club > /opt/tennis-club/data/backup_$(date +%F).sql`을 `cron`으로 `0 3 * * *` 같은 간격으로 실행.  
  * 이유: 하루 치 데이터를 잃어도 다시 복구할 수 있도록 주기적인 덤프 확보.
- 로그
  * `logs/server.out`, `logs/server.err`: pm2 혹은 systemd에서 Redirect 지정.  
  * `journalctl -u tennis-club.service`: systemd 때 추가 히스토리 확인용.  
  * 이유: 장애 발생 시 최초 로그를 빠르게 확인하기 위함.
- 장애 자동 재시작은 `pm2`가 `watch`를 통해 재기동, 또는 systemd의 `Restart=on-failure`로 서버 다운 시 자동 복구.

-## 7. 보안/네트워크
- MariaDB 접속은 내부망/VPN 또는 `ufw allow from 10.0.0.0/16 to any port 3306`처럼 방화벽 규칙으로 제한.  
  * 이유: DB는 민감한 정보+관리자 기능을 포함하고 있어 외부 노출을 최소화해야 합니다.
- 앱 포트 3000은 `ufw allow 3000/tcp`로 열고, nginx/HAProxy 같은 리버스 프록시에서 HTTPS Termination 이후 `proxy_pass`로 넘기는 구조 권장.  
  * 이유: HTTPS 인증서를 프록시에서 처리하면 Node.js 앱이 인증서 관리를 하지 않아도 되고 HTTP만 내부 통신.
- SSL 인증서는 Let's Encrypt 또는 사설 CA를 리버스 프록시(예: `/etc/nginx/sites-enabled/tennis.conf`)에 설정.
- 중요한 환경 변수(`DB_PASSWORD`)는 `/etc/systemd/system/tennis-club.service.d/override.conf` 또는 Docker secrets로 외부화하여 코드에 평문으로 남지 않게 함.
